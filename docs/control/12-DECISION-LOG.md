- Дата: 2026-02-24
- Решение: MVP = single-user.
- Причина: Быстрый запуск и личная проверка пользы.
- Последствия: Упрощённая auth, SQLite, быстрый цикл.
- Кто принял: User

- Дата: 2026-02-24
- Решение: Ценность MVP = не забывать + дожим + дисциплина/стрик + журнал.
- Причина: Это реальная ежедневная боль пользователя.
- Последствия: Напоминания и действия по целям — приоритет.
- Кто принял: User

- Дата: 2026-02-24
- Решение: "Перенести на завтра" = изменить target_date той же задачи.
- Причина: Проще для MVP.
- Последствия: История переносов хранится в events.
- Кто принял: Assistant (для MVP, можно пересмотреть)

- Дата: 2026-02-24
- Решение: Android-first на Flutter.
- Причина: Фокус на одном устройстве.
- Последствия: iOS откладывается.
- Кто принял: User

- Дата: 2026-02-24
- Решение: UI = Gemini, Logic = Claude, один репозиторий, строгий handoff.
- Причина: Разделение ролей при вайб-кодинге.
- Последствия: Обязательные control-файлы и дисциплина изменений.

- Дата: 2026-02-24
- Решение: API CONTRACT заморожен как v1 для MVP, обратная несовместимость запрещена.
- Причина: Снизить риск рассинхрона UI/Backend на старте разработки.
- Последствия: Все изменения API только через явное обновление 04-API-CONTRACT.md.
- Кто принял: User + Assistant

- Дата: 2026-02-24
- Решение: Для single-user auth используется Bearer-токен из env (`MVP_TOKEN`).
- Причина: Нужен минимальный и быстрый, но явный механизм авторизации для MVP.
- Последствия: Mobile app хранит токен локально, backend валидирует заголовок Authorization.
- Кто принял: User + Assistant

- Дата: 2026-02-24
- Решение: Переход `SNOOZE_EXPIRED` обязателен на стороне backend (лениво или фоном).
- Причина: Исключить расхождение состояния между клиентом и сервером.
- Последствия: Backend возвращает актуальный статус после sync/чтения/действий.
- Кто принял: User + Assistant
